## Task
From the user stories below, produce acceptance criteria only for what each story describes. Base every step on explicit or clearly implied behavior in the story text.

## Input
User stories (replace or follow the placeholder):
{user_stories}

## Output format
- One block per user story, in the same order as the input.
- Do not use a `Feature:` line. Do not invent product capabilities, integrations, or UI that the story does not mention.

## Rules
1. Exclude API-level or contract testing; focus on observable user- or system behavior as described in the story.
2. Write scenarios in valid Gherkin style (keywords at line start; one main Given/When/Then flow per scenario).
3. Keep steps concrete and testable, but avoid unnecessary verbosity.
4. Do not add scope beyond the user story (no extra features, no new requirements).
5. Do not infer details that are not supported by the story; if something is ambiguous, phrase the scenario at the level of certainty the story allows or note the assumption in one short line before the scenarios (only when unavoidable).
6. Produce a separate set of scenarios for each user story; do not merge stories into one scenario set.
7. Every acceptance block must contain at least one `Scenario:` with `Given` / `When` / `Then`.
8. A single user story may have multiple scenarios when the story clearly describes distinct flows or outcomes.
